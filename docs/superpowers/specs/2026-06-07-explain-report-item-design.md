# SIU 报告条目二次解释设计

## 背景

SIU 的报告会把多个 GitHub Releases 汇总成升级重点、关键修复、升级风险和逐版本摘要。部分条目可能包含用户看不懂的术语、警告或迁移提醒。新功能要让用户点击条目旁边的问号，获得更简单、更可信的二次解释。

## 目标

- 用户可以对任意报告条目点击问号，查看这条内容的简明解释。
- 解释要反查这条总结的出处，结合 release 上下文判断是否有遗漏或限定条件。
- 默认保持轻量，避免一开始就查询文档或源码造成 token 浪费。
- 二次解释必须简单易懂，总字数不超过 250 个中文字符。
- UI 继承当前极简、克制、社论式风格，不引入聊天窗口或复杂问答。

## 非目标

- 不做开放式聊天。
- 不默认查询源码。
- 不把二次解释做成完整技术文档。
- 不保证解释能替代维护者迁移指南、安全公告或人工验证。

## 用户交互

每条可解释的报告内容右侧增加一个小型问号按钮。支持的位置包括：

- 升级重点列表。
- 关键修复列表。
- 升级风险列表。
- 逐版本摘要中的单条 highlight。

点击问号后，在当前条目的下方展开解释区，而不是打开弹窗。解释区状态如下：

1. 加载中：显示一条细进度线和一句短状态，例如“正在反查出处...”。
2. 成功：展示二次解释内容。
3. 失败：展示简短错误，例如“这条暂时解释失败，可以稍后再试。”。

再次点击同一问号可收起解释区。点击其他条目的问号不强制关闭已展开解释，但实现时可以按当前页面复杂度选择是否只保留一个展开项。

## 解释内容结构

解释结果展示为四个极简段落：

```text
这是什么意思
一句话说明这条更新在说什么。

影响谁
普通用户 / 开发者 / 插件作者 / 部署者 / 基本无感。

要不要管
需要升级前检查 / 可以放心 / 只在你用到 X 时关注。

依据
来自 v1.2.3 Release：原文短摘。
```

解释区内对用户可见的自然语言合计不超过 250 个中文字符，包含“这是什么意思”“影响谁”“要不要管”“依据”四段正文。版本号、仓库名、API 名称和文件名可以出现，但也必须克制使用。

如果作者原文模糊，解释必须直接说明模糊点，例如：“作者没有说明影响范围，只能判断它和插件迁移有关。” 不允许编造确定影响。

## 分层调查策略

默认采用智能判定深度：

### Level 1：反查出处

后端先尝试把用户点击的总结条目映射回当前分析范围内的 releases：

- 匹配相关版本。
- 找到 release 原文片段。
- 保留发布时间和 release 链接。
- 检查总结是否缺少限定条件。

### Level 2：基于已有上下文解释

优先只使用现有分析上下文：

- 当前条目文本。
- 相关 release 原文片段。
- 仓库画像。
- README 摘要。
- 用户当前版本、最新版本和分析范围。

首次点击默认至少完成 Level 1 和 Level 2。

### Level 3：查文档

仅当 Level 1-2 无法解释清楚时自动触发。触发条件包括：

- 作者原文明显模糊。
- 条目包含专有名词、API、配置项、迁移动作或兼容性警告。
- 解释需要判断影响对象，但 release notes 没写清楚。

文档查询优先级：

1. README。
2. docs 目录。
3. CHANGELOG。
4. migration guide、upgrade guide、release guide 等命名相近文档。

### Level 4：查源码

源码查询不自动触发。只有解释结果标记“仍需源码确认”时，界面才可以显示一个克制的后续入口，例如“继续查源码”。

源码查询进入下一阶段实现，本次设计只预留状态和接口扩展，不在默认交互中启用。

## 后端接口

新增一个轻量接口：

```text
POST /api/explain
```

请求体建议包含：

```json
{
  "repoName": "owner/repo",
  "currentVersion": "v1.0.0",
  "latestVersion": "v1.2.0",
  "section": "coreHighlights",
  "itemText": "[体验优化] 启动速度更快",
  "itemIndex": 0,
  "releaseTag": "v1.2.0",
  "lang": "zh"
}
```

其中 `releaseTag` 对逐版本摘要更有帮助；顶层列表不一定有精确版本，可以为空。

响应体建议包含：

```json
{
  "status": "success",
  "explanation": {
    "plainMeaning": "这条是在说什么。",
    "affectedUsers": "影响谁。",
    "action": "要不要管。",
    "evidence": "来自 v1.2.0 Release：短摘。",
    "depth": "release_context",
    "needsSourceReview": false
  }
}
```

`depth` 可取值：

- `release_context`：只用了 release 和已有上下文。
- `docs`：查询了文档。
- `source_needed`：仍需源码确认，但本次不自动查询。

## 数据结构调整

当前 `UpgradeAnalysis` 只保存 AI 总结后的字符串，缺少可追溯依据。实现时需要为报告条目补充可选来源信息，或者在解释接口中重新基于分析范围反查 releases。

推荐做法：

- 保持现有展示字段不变，避免破坏已有报告渲染。
- 新增可选的 `evidenceMap` 或条目级 metadata，用于记录条目与 release 片段的关系。
- 如果历史记录中没有 metadata，解释接口仍可使用 `itemText` 和 release notes 做近似反查。

## 错误处理

- 仓库或版本范围无法解析：返回简短错误，不展开复杂技术原因。
- 找不到出处：仍可解释条目本身，但必须在依据中写明“未能定位到明确 release 原文”。
- AI 返回超长：后端二次压缩到 250 字以内。
- 文档查询失败：回退到 Level 1-2，不继续查源码。

## 成本控制

- 首次点击只发送单条 item，而不是整份报告。
- 后端只取当前版本范围内相关 release 片段，不重新发送全部 release notes。
- 文档查询只在触发条件满足时进行。
- 源码查询默认禁用。
- 同一报告条目的解释结果在前端缓存，避免重复点击重复请求。

## 测试与验证

- 构建验证：`npm run build`。
- 类型验证：`npx tsc --noEmit`。
- 接口验证：用固定请求验证 `/api/explain` 能返回 250 字内结果。
- UI 最低验证：确认问号按钮可点击、加载态出现、解释区展开、失败态可展示。
- 不对主观视觉审美做过度代劳，最终界面观感交给用户直接确认。

## 实施顺序

1. 增加解释结果类型和接口类型。
2. 增加 `/api/explain` Pages Function。
3. 在核心逻辑中实现 Level 1-2 解释。
4. 增加文档查询触发条件和 Level 3 查询。
5. 在报告条目旁增加问号按钮和行内展开区。
6. 加入前端缓存、加载态和失败态。
7. 运行构建和类型验证。
